## 📝 English Summary (Soul Injector v1.1)

**Layer 7: Team Collaboration & Task Management**

This template provides **two modes** for multi-agent collaboration:

### 🟢 Mode A: Role Workflow (Lightweight, Recommended)
- One main agent plays multiple roles (content expert, code expert, security expert, etc.)
- Shared context and memory for maximum efficiency
- No complex setup needed - works out of the box
- Best for: individual developers, small teams, daily tasks (90% use cases)

### 🔴 Mode B: True Multi-Agent (Heavyweight, Complex Scenarios)
- Each agent is a truly isolated process with independent skills and memory
- Message bus for communication between agents
- High security and scalability, but high technical threshold
- Best for: enterprise teams, complex systems, high-security requirements

**Our Recommendation**: Start with Mode A (role workflow), evolve to Mode B only when necessary.

## 🎯 两种模式：选择适合你的

Soul Injector 不假设你必须使用真正的隔离子Agent架构！我们提供 **两种模式**，你可以根据需求选择：

---

### 🟢 **模式 A：角色工作流（轻量级，推荐入门）**

> **这就是小山团队目前正在使用的模式！**

**核心原理**：一个主 Agent 扮演多个角色，共享同一套 Skill 和记忆。通过明确的职责划分和工作流，达到「团队协作」的效果。

```
┌─────────────────────────────────────────────────────────────┐
│                    主 Agent (一人饰多角)                   │
│                                                             │
│  ┌──────┐  ┌──────┐  ┌──────┐  ┌──────┐  ┌──────┐   │
│  │山小编│  │山小码│  │山小审│  │山小析│  │ ...  │   │
│  │✍️文案│  │💻代码│  │🔍安全│  │📊数据│  │      │   │
│  └──────┘  └──────┘  └──────┘  └──────┘  └──────┘   │
│                                                             │
│  ✅ 共享上下文  ✅ 共享记忆  ✅ 快速切换  ✅ 质量可控   │
└─────────────────────────────────────────────────────────────┘
```

| 优点 | 缺点 | 适用场景 |
|------|------|---------|
| ✅ 效率极高，上下文完全共享 | ❌ 不是真正的「多Agent」 | 🎯 个人开发者、小团队、日常任务 |
| ✅ 快速切换角色，无缝衔接 | ❌ 认知闭环，容易先入为主 | 🎯 快速原型、敏捷开发 |
| ✅ 质量可控，输出一致性高 | ❌ 没有真正的观点碰撞 | 🎯 大多数日常场景 |
| ✅ 无需额外配置，开箱即用 | ❌ 不适合需要绝对隔离 | 🎯 90% 的普通用户 |

---

### 🔴 **模式 B：真正多Agent（重量级，复杂场景用）**

**核心原理**：每个 Agent 是真正独立、隔离的进程，有自己的 Skill 配置、独立的记忆空间、甚至运行在不同的容器/机器上。通过消息总线进行通信。

```
┌─────────────────────────────────────────────────────────────┐
│                    任务调度器 (Scheduler)                   │
└──────────────────────────────┬──────────────────────────────┘
                               │ 消息总线
        ┌──────────────────────┼──────────────────────┐
        │                      │                      │
┌───────▼───────┐   ┌──────▼───────┐   ┌──────▼───────┐
│  内容专家    │   │  代码专家    │   │  数据专家    │
│  独立进程    │   │  独立进程    │   │  独立进程    │
│  独立Skill   │   │  独立Skill   │   │  独立Skill   │
│  独立记忆    │   │  独立记忆    │   │  独立记忆    │
└──────────────────┘   └──────────────────┘   └──────────────────┘
```

| 优点 | 缺点 | 适用场景 |
|------|------|---------|
| ✅ 真正的观点碰撞，避免认知闭环 | ❌ 配置复杂，技术门槛高 | 🎯 大型企业、专业团队 |
| ✅ 真正的专业分工，术业专攻 | ❌ 上下文切换成本高 | 🎯 复杂系统开发 |
| ✅ 安全隔离，敏感数据不泄露 | ❌ 同步复杂，需要消息总线 | 🎯 高安全要求场景 |
| ✅ 可扩展，支持横向扩展 | ❌ 调试困难，问题定位难 | 🎯 超大规模项目 |

## 🚀 任务分发脚本 `task-dispatcher.sh`

```bash
#!/bin/bash
# ============================================================
# Soul Injector - 子 Agent 任务分发器
# ============================================================
# 用法: bash task-dispatcher.sh [任务类型] "[任务描述]"
#
# 任务类型: content / code / data / security / auto
# ============================================================

set -e

SCRIPT_DIR="$(cd "$(dirname "${BASH_SOURCE[0]}")" && pwd)"
REGISTRY_FILE="$SCRIPT_DIR/registry.md"
STATUS_FILE="$SCRIPT_DIR/status-tracker.md"

# 颜色定义
RED='\033[0;31m'
GREEN='\033[0;32m'
YELLOW='\033[1;33m'
BLUE='\033[1;34m'
NC='\033[0m'

# 生成唯一任务 ID
generate_task_id() {
    echo "task-$(date +%Y%m%d-%H%M%S)-$(openssl rand -hex 4)"
}

# ============================================================
# 步骤 1: 智能匹配最合适的子 Agent
# ============================================================
match_agent() {
    local task_type="$1"
    
    echo -e "${BLUE}🔍 智能匹配子 Agent...${NC}"
    echo ""
    
    case "$task_type" in
        content|文档|写作|文案)
            echo "✅ 匹配到: 山小编 ✍️ (内容专家)"
            echo "   擅长: 文档撰写、内容优化、文案创作"
            AGENT_NAME="山小编"
            AGENT_EMOJI="✍️"
            ;;
        code|代码|开发|调试)
            echo "✅ 匹配到: 山小码 💻 (代码专家)"
            echo "   擅长: 代码开发、调试修复、架构设计"
            AGENT_NAME="山小码"
            AGENT_EMOJI="💻"
            ;;
        data|数据|分析|可视化)
            echo "✅ 匹配到: 山小析 📊 (数据专家)"
            echo "   擅长: 数据分析、统计建模、可视化"
            AGENT_NAME="山小析"
            AGENT_EMOJI="📊"
            ;;
        security|安全|审计|漏洞)
            echo "✅ 匹配到: 山小审 🔍 (安全专家)"
            echo "   擅长: 安全审计、漏洞扫描、合规检查"
            AGENT_NAME="山小审"
            AGENT_EMOJI="🔍"
            ;;
        devops|运维|部署|监控)
            echo "✅ 匹配到: 山小运 🚀 (运维专家)"
            echo "   擅长: 部署运维、CI/CD、监控告警"
            AGENT_NAME="山小运"
            AGENT_EMOJI="🚀"
            ;;
        *)
            echo "🟡 未指定类型，自动匹配..."
            echo "✅ 匹配到: 山小编 ✍️ (默认内容专家)"
            AGENT_NAME="山小编"
            AGENT_EMOJI="✍️"
            ;;
    esac
}

# ============================================================
# 步骤 2: 记录任务到状态追踪板
# ============================================================
record_task() {
    local task_id="$1"
    local task_type="$2"
    local task_desc="$3"
    local agent_name="$4"
    local agent_emoji="$5"
    local timestamp=$(date "+%Y-%m-%d %H:%M:%S")
    
    echo ""
    echo -e "${BLUE}📝 记录任务状态...${NC}"
    
    # 追加到状态追踪板
    cat >> "$STATUS_FILE" << EOF

---

### $task_id $agent_emoji $agent_name

| 字段 | 值 |
|------|-----|
| **任务 ID** | \`$task_id\` |
| **任务类型** | $task_type |
| **任务描述** | $task_desc |
| **负责人** | $agent_name $agent_emoji |
| **创建时间** | $timestamp |
| **状态** | 🟡 进行中 |
| **开始时间** | $timestamp |
| **预计完成** | - |
| **实际完成** | - |
| **结果摘要** | - |

EOF
    
    echo "✅ 任务已记录，ID: $task_id"
}

# ============================================================
# 步骤 3: 执行任务（模拟子 Agent 工作）
# ============================================================
execute_task() {
    local task_id="$1"
    local task_type="$2"
    local task_desc="$3"
    local agent_name="$4"
    local agent_emoji="$5"
    
    echo ""
    echo -e "${BLUE}🚀 $agent_name $agent_emoji 开始执行任务...${NC}"
    echo ""
    echo "任务: $task_desc"
    echo ""
    
    # 模拟执行过程
    echo "⏳ 执行中..."
    sleep 2
    echo "✅ 子任务 1 完成..."
    sleep 1
    echo "✅ 子任务 2 完成..."
    sleep 1
    echo "✅ 子任务 3 完成..."
    sleep 1
    
    local finish_time=$(date "+%Y-%m-%d %H:%M:%S")
    
    echo ""
    echo -e "${GREEN}🎉 任务完成！${NC}"
    echo "   完成时间: $finish_time"
    
    # 更新状态追踪板
    sed -i "s/| **状态** | 🟡 进行中 |/| **状态** | ✅ 已完成 |/" "$STATUS_FILE"
    sed -i "s/| **实际完成** | - |/| **实际完成** | $finish_time |/" "$STATUS_FILE"
}

# ============================================================
# 步骤 4: 结果汇总
# ============================================================
aggregate_result() {
    local task_id="$1"
    
    echo ""
    echo -e "${BLUE}📊 结果汇总中...${NC}"
    echo ""
    
    echo "✅ 任务 $task_id 已完成"
    echo ""
    echo "📋 完整任务结果已记录到状态追踪板"
    echo "🔗 查看详情: $STATUS_FILE"
}

# ============================================================
# 主程序
# ============================================================
main() {
    local task_type="${1:-auto}"
    local task_desc="$2"
    
    if [ -z "$task_desc" ]; then
        echo -e "${RED}❌ 请提供任务描述${NC}"
        echo "用法: bash task-dispatcher.sh [任务类型] \"任务描述\""
        echo ""
        echo "任务类型: content / code / data / security / auto"
        exit 1
    fi
    
    clear
    echo "╔══════════════════════════════════════════════════════════════╗"
    echo "║           Soul Injector - 子 Agent 任务分发器              ║"
    echo "╚══════════════════════════════════════════════════════════════╝"
    echo ""
    
    # 步骤 1: 匹配 Agent
    match_agent "$task_type"
    
    # 步骤 2: 生成任务 ID 并记录
    local task_id=$(generate_task_id)
    record_task "$task_id" "$task_type" "$task_desc" "$AGENT_NAME" "$AGENT_EMOJI"
    
    # 步骤 3: 执行任务
    execute_task "$task_id" "$task_type" "$task_desc" "$AGENT_NAME" "$AGENT_EMOJI"
    
    # 步骤 4: 结果汇总
    aggregate_result "$task_id"
    
    echo ""
    echo -e "${GREEN}✅ 全部完成！${NC}"
}

main "$@"
```

## 💡 最佳实践

### 1. 专业分工原则

- 每个子 Agent 只专注 1-2 个领域
- 不要让一个子 Agent 什么都做
- 复杂任务拆分成多个子任务，分发给多个子 Agent

### 2. 状态透明原则

- 所有任务状态必须记录在追踪板
- 任务完成后自动汇总结果
- 阻塞任务及时上报主 Agent

### 3. 结果可追溯原则

- 每个任务有唯一 ID
- 所有变更都有时间戳
- 结果和原始任务关联
